SECTION 1: WHAT PS6 IS ACTUALLY ASKING FOR
================================================================================

Problem Statement 6 — "AI for Digital Public Safety: Defeating Counterfeiting,
Fraud & Digital Arrest Scams"

The PS lists 5 illustrative components. We build a focused slice:

COMPONENT A — Fraud Network Graph Intelligence (PRIMARY, REUSED)
  Graph AI that analyses transaction data and account linkages to map
  coordinated fraud, mule networks, and ring/chain patterns into an
  investigation package. This is FundFlow's graph engine + a new agentic
  layer on top.
  NOTE: The PS wording mentions "court-admissible evidence." We do NOT claim
  that. An AI-drafted report is an INTERNAL COMPLIANCE ARTIFACT (a draft STR),
  not evidence and not court-admissible. We frame the output as a draft
  Suspicious Transaction Report for a human analyst to review and file.

COMPONENT B — Agentic STR Investigator (PRIMARY — this IS the FYP MVP)
  An LLM agent (LangGraph) that, given a flagged alert or suspicious account,
  autonomously plans an investigation by calling tools (transaction history,
  graph analysis, risk scoring, regulation lookup, typology match) and outputs
  a CITED draft Suspicious Transaction Report (STR) aligned to RBI / FIU-IND
  guidance, with FATF / FinCEN as supporting international context.
  → This IS the FYP. The hackathon is the MVP. The semester is the polish.

COMPONENT C — Digital Arrest Scam Detector (DEMO ADD-ON, LOWEST PRIORITY)
  A few-shot LLM classifier that takes a call transcript and flags likely
  digital-arrest / fake-officer / KYC-freeze scams before money moves.
  → Cheap to build, strong demo appeal (₹1,776 crore lost in first 9 months
    of 2024 — extremely topical for judges). But it is a SEPARATE mini-project
    that shares no pipeline with the AML agent. Treat it as garnish, not a
    co-equal module. Build it LAST, and only if PART A is already solid.

TERMINOLOGY NOTE (get this right — interviewers WILL test it):
  - India: STR (Suspicious Transaction Report), filed with FIU-IND under PMLA.
  - US   : SAR (Suspicious Activity Report), filed with FinCEN.
  Because our regulatory framing is India-first (RBI, PMLA, digital-arrest
  scams), the primary term is STR. Use "SAR" only when explicitly discussing
  the US/FinCEN framework. Do not use them interchangeably.

WHAT WE ARE NOT BUILDING (see Section 9): counterfeit currency CV, geospatial
crime intelligence, WhatsApp/IVR citizen shield.

JUDGING CRITERIA (design every decision against this):
  Innovation        25%  ← Agentic RAG + autonomous STR drafting is the hook
  Business Impact   25%  ← Real AML use case + ₹1,776cr digital-arrest angle
  Technical Excel.  20%  ← Real reg corpus, SHAP, guardrails, measured eval
  Scalability       15%  ← FastAPI, modular agent tools, vector store
  User Experience   15%  ← Investigation console UI + a clean demo video

SECTION 2: WHAT WE ARE ACTUALLY BUILDING

THE PLATFORM: "SENTINEL AI — Fraud Network Intelligence & AML Investigation"

MODULE 1: FRAUD NETWORK INTELLIGENCE ENGINE (reused from FundFlow, AS-IS)
  - Transaction graph: fund flows, mule accounts, ring/chain patterns
  - XGBoost risk scoring per transaction (score_risk tool)
  - SHAP explainability per flagged alert
  - Mule/ring detection (ring_id, chain_id, fan_out_ratio, mule_score)
  - Exposed via FastAPI as callable tools for the agent
  → For the hackathon this runs on FundFlow's EXISTING PaySim-trained model.
    Do not retrain for the MVP. (See Section 3 and Section 5.)

MODULE 2: AGENTIC AML INVESTIGATOR (the star of the show)
  Given a flagged alert or account ID, the agent:
    Step 1 → get_transaction_history(account_id) — from SQLite
    Step 2 → get_transaction_graph(account_id)   — NetworkX subgraph
    Step 3 → score_risk(transaction)             — FundFlow XGBoost engine
    Step 4 → search_regulations(query)           — RAG over real FATF/FinCEN/RBI
    Step 5 → detect_typology(pattern)            — structuring/layering/smurfing
    Step 6 → synthesizes a CITED draft STR with risk score + rationale

  IMPORTANT: If detect_typology is not finished by the hackathon deadline, the
  agent ships with 4 tools (history, graph, score_risk, search_regulations).
  Do not claim "5 tools" unless all 5 are actually wired in. See PART A.

MODULE 3: DIGITAL ARREST SCAM CLASSIFIER (demo add-on, build LAST)
  Few-shot GPT-4o-mini classifier over a pasted call transcript. Outputs:
  scam probability, scam type, trigger phrases, recommended citizen action.
  HONESTY: it is few-shot prompted on SYNTHETIC scripts and demoed on
  SYNTHETIC scripts — that is circular and proves little about real scams.
  Present it as a demonstrator, NOT a validated/trained model. Do not report
  accuracy numbers for it.

SECTION 5: HOW FUNDFLOW IS USED AS A TOOL IN PS6
================================================================================

The FundFlow XGBoost engine becomes the score_risk tool in the agent.

score_risk(transaction_dict) inside tools.py:
  1. Takes a transaction dict (same format as FundFlow gateway)
  2. Calls models/predictor.py -> predict_single()
  3. Returns { fraud_probability, risk_tier, top_features (SHAP) }
  The agent cites this in the STR, e.g.:
    "XGBoost risk score: 0.89 (CRITICAL). Top SHAP drivers: amount_deviation
     (+0.42), sender_txn_count_1h (+0.31), fan_out_ratio (+0.28) — consistent
     with a velocity-burst pattern."
  (Use REAL scores from your model at demo time, not placeholder numbers.)

MODEL / DATASET NOTE:
  For the hackathon, use xgboost_fraud.pkl AS-IS on PaySim. Do NOT retrain.
  Retraining on AMLSim (different columns => re-derived features) is a PART B
  task. Keeping the existing model is what makes Module 1 "free" on day one.

get_transaction_graph: graph/fund_flow.py + graph/mule_detector.py return a
  NetworkX subgraph for an account -> mule_score, ring_id, chain_id,
  fan_out_ratio, 1st/2nd-degree connections -> evidence in the STR.

get_transaction_history: SQL query against the SQLite DB -> last N transactions
  for an account (amount, counterparty, timestamp) -> velocity/anomaly signals.

search_regulations: the RAG tool. Query -> top-3 cited passages from the
  FATF/FinCEN/RBI corpus with document name + page. The agent MUST cite; uncited
  regulatory claims are rejected by guardrails.

detect_typology: rule-based + graph-derived matcher:
  - Structuring: multiple transactions just below reporting thresholds
  - Layering:    sequential multi-hop movement (chain_id present)
  - Smurfing:    one destination receiving from many sources (fan-out)
  - Round-tripping: circular flow (ring_id present)
  Reuses ring_detector.py + mule_detector.py. If not done by the deadline, the
  MVP ships with 4 tools (see Section 2).

SECTION 11: INTERVIEW DEFENSE POINTS
================================================================================

Be able to answer these cold:

1. Why agentic over a single RAG call?
   Single RAG = one retrieval, one answer. Agentic = multiple tool calls,
   reasoning between steps, accumulating evidence before drafting. A draft STR
   needs transaction history + graph patterns + regulations cross-referenced —
   inherently multi-step.

2. How does your RAG pipeline work?
   PDFs chunked to ~500 tokens (50 overlap) -> embedded with
   sentence-transformers/all-MiniLM -> stored in Chroma -> cosine-similarity
   retrieval -> top-3 passages returned with document name + page.

3. What are your eval metrics and what do they mean?
   Faithfulness: answer claims are supported by retrieved passages.
   Context relevance: retrieved passages are actually relevant to the query.
   Answer relevance: the answer addresses the question.
   Context recall: all relevant passages were retrieved.
   (Quote only numbers you actually measured.)

4. How does FundFlow integrate?
   models/predictor.py -> predict_single() is wrapped as the score_risk agent
   tool; it returns fraud_probability + SHAP top features, cited in the STR.

5. Why synthetic transaction data?
   Real bank transaction data is legally inaccessible (privacy law). The AML
   research community uses synthetic sets (PaySim, IBM AMLSim, SAML-D); we
   disclose this. The RAG corpus (FATF/FinCEN/RBI) is genuine regulation — that
   layer is real. Elliptic++ is available as a real-graph extension.

6. STR or SAR — which and why?
   STR is the Indian term (filed with FIU-IND under PMLA); SAR is the US term
   (FinCEN). Our framing is India-first, so the output is a draft STR; FATF and
   FinCEN are supporting international context.

7. Is the output court-admissible evidence?
   No. It is an internal compliance DRAFT for a human analyst to review and
   file. We never claim it is evidence.

8. What are your guardrails?
   (a) Citation enforcement: no regulatory claim without a supporting retrieved
   passage. (b) Evidence threshold: draft only generated after >=3 tool calls
   and risk score above the flag threshold. (c) A string-match hallucination
   check for regulatory names/dates not in retrieved passages — a heuristic,
   not a guarantee (be honest about its false-positive/negative limits).

9. The digital-arrest classifier — how good is it?
   It is a few-shot demonstrator on synthetic scripts; training and testing on
   the same synthetic distribution is circular, so I do not report accuracy for
   it. It illustrates the citizen-protection angle; validating it needs real
   data.
