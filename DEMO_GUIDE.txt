------------------------------------------------------------

  - Charts: fraud distribution, risk breakdown

   6.3 million transactions processed through our XGBoost
   fraud engine. We have 264 active alerts across all risk

  - Click "Start Simulation"
  - Watch 5-10 transactions appear in the live feed
  - Point out: red = HIGH/CRITICAL, green = LOW
  - Click "Why?" on a red transaction to show SHAP modal

   'Why?', the system shows the exact SHAP feature contributions
   that caused this flag — this is model explainability."

  - Stop the simulator

OPTIONAL: If the judge asks follow-up questions
============================================================

Q: "Why PaySim dataset?"
A: "PaySim provides realistic fraud labels from real banking
    patterns. We don't use PaySim's raw features — we compute
    49 behavioral features from the transaction graph using our
    own engineering pipeline."

Q: "Can you explain the features?"
A: Open features/engineering.py or FEATURE_REFERENCE.md

Q: "Does KYC matter?"
A: "KYC alone is weak — C1953680528 has biometric KYC and
    100% fraud probability. KYC only adds value when combined
    with behavioral anomalies via our kyc_risk_flag feature."

Q: "What about production deployment?"
A: "Our prototype uses SQLite and NetworkX for zero-dependency
    deployment. Production would replace these with Kafka for
    real-time streaming, Redis for feature cache, Neo4j for
    graph persistence, and PostgreSQL for transaction storage.
    The XGBoost pipeline stays identical."

Q: "How fast is it?"
A: "Point at the latency measurement — typically 20-40ms on
    a laptop. In production with Redis, this drops to <15ms."
