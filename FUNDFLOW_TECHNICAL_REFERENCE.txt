================================================================================
SECTION 1: DATABASE SCHEMA (fundflow.db — SQLite, 205 MB)
================================================================================

The database has 4 tables. For PS6, only `transactions` and `account_profiles`
matter. The rest are FundFlow-specific and are not reused.

TABLE: transactions
  Row count : 499,196
  Purpose   : One row per financial transaction (PaySim-derived, India-mapped)

  Columns:
    txn_id                   TEXT    — Unique ID, format "TXN_00000000"
    timestamp                TEXT    — "2026-03-01 01:00:15" (ISO datetime string)
    sender_account           TEXT    — Account ID, format "C1828508781" (customer)
                                       or "M1273958371" (merchant)
    receiver_account         TEXT    — Same format
    amount                   REAL    — Transaction amount in INR
    txn_type                 TEXT    — "UPI", "NEFT", "ATM", "DEPOSIT", "IMPS"
    sender_branch            TEXT    — "BR_MUMBAI_001", "BR_DELHI_002", etc.
    receiver_branch          TEXT    — Same format
    sender_balance_before    REAL    — Balance before txn (can be 0.0 for merchants)
    sender_balance_after     REAL    — Balance after txn
    receiver_balance_before  REAL    — (can be 0.0 for merchants)
    receiver_balance_after   REAL
    channel                  TEXT    — "mobile", "internet", "branch"
    is_fraud                 INT     — Ground truth: 1=fraud, 0=legit
    is_flagged_fraud         INT     — Model flag (may differ from ground truth)
    step                     INT     — PaySim simulation step (1 step ≈ 1 hour)
    fraud_probability        REAL    — XGBoost score (0.0 to 1.0)
    risk_score               REAL    — Same as fraud_probability in current version
    risk_tier                TEXT    — "LOW", "MEDIUM", "HIGH", "CRITICAL"

  Sample account ID formats:
    "C1828508781" — Customer accounts (C prefix)
    "M1273958371" — Merchant accounts (M prefix)

TABLE: account_profiles
  Row count : 875,215
  Purpose   : India-specific KYC/credit data per account (used by feature eng.)

  Columns:
    account_id        TEXT    — Matches sender_account/receiver_account in transactions
    vpa               TEXT    — Virtual Payment Address (UPI handle, e.g. "user@oksbi")
    bank_handle       TEXT    — "oksbi", "ybl", "ibl", "axl", "apl", "paytm", etc.
    kyc_type          TEXT    — "biometric", "otp_ekyc", "video_kyc", "paper"
    account_age_days  INT     — How old the account is in days
    credit_score      INT     — CIBIL-style score (300–900 range)

TABLE: alerts  (FundFlow-specific — not reused in PS6)
  Columns: alert_id, timestamp, severity, alert_type, accounts_involved,
           total_amount, risk_score, description, recommended_action,
           status, evidence

TABLE: cases  (FundFlow-specific — not reused in PS6)
  Columns: case_id, status, priority, created_at, updated_at, assigned_to,
           linked_alerts, linked_accounts, total_exposure, timeline, notes, evidence

HOW TO QUERY THE DB IN PS6:

  import sqlite3
  import pandas as pd

  DB_PATH = "/path/to/fundflow.db"   # from .env: PS6_DB_PATH

  # Get last N transactions for an account (get_transaction_history tool)
  conn = sqlite3.connect(DB_PATH)
  df = pd.read_sql_query(
      "SELECT * FROM transactions WHERE sender_account = ? OR receiver_account = ? "
      "ORDER BY timestamp DESC LIMIT ?",
      conn, params=(account_id, account_id, n)
  )
  conn.close()

  # Get KYC profile for an account
  conn = sqlite3.connect(DB_PATH)
  row = pd.read_sql_query(
      "SELECT * FROM account_profiles WHERE account_id = ?",
      conn, params=(account_id,)
  ).to_dict('records')
  conn.close()


================================================================================
SECTION 2: models/predictor.py — The score_risk Tool
================================================================================

FILE: models/predictor.py
PURPOSE: Load the trained XGBoost model and run fraud predictions.

MAIN FUNCTION: predict_single()

  Signature:
    predict_single(
        txn: dict,                        # REQUIRED — transaction dict (see below)
        account_history: pd.DataFrame,    # OPTIONAL — recent sender transactions
        graph_features: dict,             # OPTIONAL — precomputed graph signals
        india_extras: dict                # OPTIONAL — KYC/CIBIL data
    ) -> dict

  txn dict keys (all optional except amount and sender_account):
    "txn_id"           : str   — unique ID for this transaction
    "amount"           : float — transaction amount in INR
    "timestamp"        : str   — ISO datetime string
    "sender_account"   : str   — sender account ID
    "receiver_account" : str   — receiver account ID
    "txn_type"         : str   — "UPI", "NEFT", "ATM", "DEPOSIT", "IMPS"
    "channel"          : str   — "mobile", "internet", "branch"
    "sender_branch"    : str   — branch code
    "receiver_branch"  : str   — branch code

  account_history: pd.DataFrame with columns from the `transactions` table.
    Pass the last N transactions for the same sender from the DB.
    If None → behavioral features (amount_deviation, velocity) default to safe values.

  graph_features: dict, format {account_id: {feature: value}}
    Get this from features/graph_features.py -> load_graph_features()
    If None → all graph-derived features (mule_score, in_ring, etc.) default to 0.

  india_extras: dict, format {account_id: {kyc_type, account_age_days, credit_score}}
    Get from account_profiles table or scripts/generate_india_extras.py
    If None → KYC features default to 0 (biometric, 365 days, score 700)

  Returns dict:
    {
      "fraud_probability" : float  — 0.0 to 1.0 (0.7+ = flagged)
      "fraud_label"       : int    — 1 if fraud_probability >= threshold, else 0
      "decision_threshold": float  — currently 0.7
      "risk_tier"         : str    — "LOW" / "MEDIUM" / "HIGH" / "CRITICAL"
      "top_features"      : list   — [{"feature": str, "contribution": float}, ...]
                                     Top 5 features by contribution magnitude
      "raw_features"      : dict   — all 48 engineered feature values
    }

SIMPLEST USAGE IN PS6 tools.py (score_risk tool):

  import sys, os
  sys.path.insert(0, "/path/to/ps6/folder")   # so imports resolve
  from models.predictor import predict_single
  import sqlite3, pandas as pd

  def score_risk(account_id: str, transaction: dict) -> dict:
      conn = sqlite3.connect(os.environ["PS6_DB_PATH"])
      history = pd.read_sql_query(
          "SELECT * FROM transactions WHERE sender_account = ? "
          "ORDER BY timestamp DESC LIMIT 50",
          conn, params=(account_id,)
      )
      conn.close()

      result = predict_single(
          txn=transaction,
          account_history=history if len(history) > 0 else None
          # graph_features and india_extras can be None for MVP
      )
      return {
          "fraud_probability": result["fraud_probability"],
          "risk_tier":         result["risk_tier"],
          "top_features":      result["top_features"],  # SHAP-lite top 5
      }

NOTE ON MODEL FILE PATH:
  predictor.py uses config.MODEL_DIR to find xgboost_fraud.pkl.
  config.MODEL_DIR = BASE_DIR/models/saved where BASE_DIR = the folder containing config.py.
  As long as the PS6 folder structure is preserved (config.py at root,
  models/saved/xgboost_fraud.pkl exists), the model loads automatically.

SECTION 3: features/engineering.py — Feature Engineering Pipeline
================================================================================

FILE: features/engineering.py
PURPOSE: Convert raw transaction dicts into the 48-feature vector the model expects.

TWO KEY FUNCTIONS:

  engineer_single(txn, account_history, graph_features, india_extras) -> dict
    Used for real-time inference (one transaction at a time).
    Same signature as predict_single above.
    Returns a plain dict of {feature_name: value} — 48 features total.
    predict_single() calls this internally — you do NOT call it directly
    unless you need the raw features for another purpose.

  engineer_features(df, graph_features, india_extras) -> pd.DataFrame
    Used for batch processing (training, bulk scoring).
    Takes a full DataFrame of transactions and returns the DataFrame with
    all 48 feature columns added.

  get_feature_columns() -> list
    Returns the ordered list of all 48 feature column names.
    The model requires exactly these 48 columns in this order.

THE 48 FEATURE COLUMNS (in exact order):

  Group 1 — Transaction-level (6):
    amount_log, hour_of_day, day_of_week, is_weekend, is_night, is_cross_branch

  Group 2 — Amount type (8):
    amount_bucket, type_NEFT, type_UPI, type_ATM, type_DEPOSIT, type_IMPS,
    channel_mobile, channel_internet

  Group 3 — Receiver / cross-bank (3):
    is_new_receiver, is_cross_bank_upi, upi_new_recv_risk

  Group 4 — Aggregate stats (2):
    receiver_total_recv_count, sender_total_unique_receivers

  Group 5 — Rolling velocity / behavioral (8):
    sender_txn_count_1h, sender_txn_count_24h, sender_avg_amount,
    sender_std_amount, amount_deviation, sender_unique_receivers_1h,
    time_since_last_txn_min, hour_velocity_ratio

  Group 6 — Rule-based / India thresholds (10):
    near_any_threshold, near_50k_threshold, near_100k_threshold,
    near_1m_threshold, is_round_10k, is_round_1k,
    high_velocity_1h, high_velocity_24h, rapid_succession, amount_gt_5x_avg

  Group 7 — Graph features — sender (4):
    sender_mule_score, sender_in_ring, sender_passthrough_ratio,
    sender_is_new_account

  Group 8 — Graph features — receiver (6):
    receiver_mule_score, receiver_in_ring, receiver_unique_senders_total,
    receiver_is_pure_receiver, receiver_is_suspected_mule, receiver_is_new_account

  Group 9 — India-specific (2):
    kyc_risk_flag, cibil_high_txn_flag

WHAT HAPPENS IF OPTIONAL ARGS ARE MISSING:
  account_history=None → velocity features default to 1, amount_deviation to 0.0
                          (safe, slightly under-scores risk — acceptable for MVP)
  graph_features=None  → all graph columns default to 0.0
  india_extras=None    → kyc_risk_flag=0, cibil_high_txn_flag=0


================================================================================
SECTION 4: graph/fund_flow.py — Transaction Graph Engine
================================================================================

FILE: graph/fund_flow.py
CLASS: FundFlowGraph

PURPOSE: Builds a directed multigraph (NetworkX MultiDiGraph) where:
  Nodes = bank account IDs
  Edges = individual transactions (with amount, timestamp, fraud_prob)

HOW TO USE IN PS6:

  from graph.fund_flow import FundFlowGraph
  import pandas as pd, sqlite3

  # Build the graph once at startup (load a sample — 499K rows takes ~10s)
  conn = sqlite3.connect(DB_PATH)
  df = pd.read_sql_query("SELECT * FROM transactions LIMIT 100000", conn)
  conn.close()

  ffg = FundFlowGraph()
  ffg.build_from_df(df)

KEY METHODS:

  build_from_df(df: pd.DataFrame) -> self
    Required columns: txn_id, sender_account, receiver_account, amount,
                      timestamp, fraud_probability, is_fraud, txn_type
    Builds the graph in-memory. For 100K rows takes ~5-10 seconds.

  trace_fund_flow(start_account, max_hops=6, time_window_hours=24) -> dict
    Traces where money flows FROM start_account, up to max_hops deep.
    Only follows causally-ordered edges (receiver must spend AFTER receiving).
    Returns:
      {
        "paths"  : list of paths (each = list of (account, amount, timestamp) tuples)
        "nodes"  : list of account IDs involved
        "edges"  : list of {from, to, txn_id, amount, timestamp, fraud_prob,
                             is_fraud, txn_type, hop}
        "summary": {start_account, total_hops, nodes_involved, total_amount,
                    time_span_min, fraud_edges}
      }
    → This is the get_transaction_graph tool in PS6.

  get_account_profile(account: str) -> dict
    Returns node-level stats:
      {account, total_sent, total_received, txn_count, in_degree, out_degree,
       unique_senders, unique_receivers, max_fraud_prob, net_flow}
    net_flow = total_received - total_sent
    Negative net_flow = more sent than received (typical mule behavior)

  find_shortest_path(source, target) -> dict
    Checks if money flowed from source to target via any path.
    Returns: {exists: bool, path: list, hops: int}

  get_graph_stats() -> dict
    Returns: {total_nodes, total_edges, is_connected}

ACCESSING THE RAW NETWORKX GRAPH:
  ffg.G  — the nx.MultiDiGraph object
  Pass this to mule_detector.compute_mule_scores() and ring_detector.find_cycles()


================================================================================
SECTION 5: graph/mule_detector.py — Mule Account Detection
================================================================================

FILE: graph/mule_detector.py
PURPOSE: Score every account for mule-like behavior.

MAIN FUNCTION: compute_mule_scores()

  Signature:
    compute_mule_scores(
        G: nx.MultiDiGraph,    # from FundFlowGraph.G
        df: pd.DataFrame,      # raw transactions DataFrame (needs 'step' column)
        kyc_data: dict = None  # {account_id: {kyc_type, account_age_days}}
    ) -> pd.DataFrame

  Returns DataFrame with columns:
    account              — account ID
    mule_score           — 0.0 to 1.0 (>=0.6 = suspected mule)
    passthrough_ratio    — total_sent / total_received (close to 1.0 = pass-through)
    avg_fwd_delay_min    — minutes between first receive and first forward send
    unique_senders       — number of unique accounts sending TO this account
    unique_receivers     — number of unique accounts this account sends TO
    total_received       — total INR received
    total_sent           — total INR sent
    amount_cluster_score — how tightly clustered the transaction amounts are
    kyc_type             — from kyc_data if provided
    account_age_days     — from kyc_data if provided
    is_suspected_mule    — 1 if mule_score >= 0.6, else 0

MULE SCORE FORMULA (5 signals weighted sum, capped at 1.0):
  S1 = passthrough_ratio × 0.30
  S2 = forward_delay_score × 0.25  (1.0 if <30min, 0.5 if <120min, 0 otherwise)
  S3 = min(unique_senders/10, 1.0) × 0.20
  S4 = new_account_score × 0.15    (1.0 if step>600, 0.5 if step>300, else 0)
  S5 = amount_cluster_score × 0.10
  S6 = 0.15 if kyc_type='otp_ekyc' AND account_age_days<90 else 0.0

SECONDARY FUNCTION: get_mule_network()

  Signature:
    get_mule_network(
        mule_df: pd.DataFrame,   # output of compute_mule_scores()
        G: nx.MultiDiGraph,
        threshold: float = 0.6
    ) -> dict

  Returns:
    {
      "nodes": [{id, label, mule_score, color}],  # color: red if >0.8, amber otherwise
      "edges": [{from, to, amount, txn_id}],
      "suspected_mules": int
    }
  → Used for graph visualization in the Streamlit console.

HOW TO GET mule_score FOR A SPECIFIC ACCOUNT (for score_risk tool):
  mule_df = compute_mule_scores(ffg.G, transactions_df)
  score_row = mule_df[mule_df['account'] == account_id]
  mule_score = float(score_row['mule_score'].values[0]) if len(score_row) > 0 else 0.0

SECTION 6: graph/ring_detector.py — Circular Flow Detection
================================================================================

FILE: graph/ring_detector.py
PURPOSE: Find cyclic fund flows (A→B→C→A) in the transaction graph.

MAIN FUNCTION: find_cycles()

  Signature:
    find_cycles(
        G: nx.MultiDiGraph,
        max_length: int = 6,           # max accounts in the ring
        time_window_hours: int = 48    # all edges must be within this window
    ) -> list

  Returns list of ring dicts (sorted by risk_score descending):
    {
      "ring_id"      : "RING_0000"
      "accounts"     : [account_id, ...]     # all accounts in the cycle
      "ring_size"    : int                   # number of accounts
      "edges"        : [{from, to, amount, timestamp, txn_id}]
      "total_amount" : float
      "time_span_hrs": float
      "risk_score"   : float  (0.0 to 1.0)
    }

RING RISK SCORE FORMULA:
  Base:  +30 (any cycle)
  Short: +20 if ring_size <= 3
  Fast:  +25 if time_span_hrs < 2  |  +10 if < 6
  Tight: +15 if amount CV (std/mean) < 0.05 (near-identical amounts)
  Final: min(total/100, 1.0)

SECONDARY FUNCTIONS:
  find_suspicious_clusters(G, min_degree=5) -> list
    Finds hub accounts with unusually high total connections.
    Returns top 50 by total_degree: [{account, in_degree, out_degree,
    total_degree, total_sent, total_received, max_fraud_prob}]

  get_ring_summary(rings: list) -> dict
    Returns: {total_rings, high_risk_rings, total_exposure, avg_ring_size}

HOW TO CHECK IF AN ACCOUNT IS IN A RING (for agent tool):
  rings = find_cycles(ffg.G)
  account_rings = [r for r in rings if account_id in r['accounts']]
  in_ring = len(account_rings) > 0
  ring_ids = [r['ring_id'] for r in account_rings]

SECTION 7: scoring/composite.py — 6-Layer Risk Engine
================================================================================

FILE: scoring/composite.py
PURPOSE: Combine 6 risk signals into one composite score using the formula:
  CompositeScore = 0.7 × max(L1..L6) + 0.3 × weighted_average(L1..L6)

This prevents any single benign signal from diluting a critical one.
The 6 layers and their weights:

  L1 — Location Anomaly    (weight: 0.12)
    Scores based on cross-bank transfer and cooperative bank risk.
    Input: txn_type, sender_branch, receiver_branch, account risk scores

  L2 — Channel Trust       (weight: 0.10)
    Risk ranking: ATM/CASH 0.55 > UPI 0.25 > IMPS 0.22 > NEFT 0.18 > NET_BANKING 0.15
    Input: txn_type, channel

  L3 — Behavioral Pattern  (weight: 0.18)
    Amount vs sender historical average. Ratios: ×20→0.95, ×10→0.85, etc.
    Also fires on PMLA structuring thresholds.
    Input: amount, sender_avg_amount

  L4 — ML Probability      (weight: 0.35 — highest)
    Raw XGBoost output (fraud_probability from predictor.py)

  L5 — Network Graph       (weight: 0.15)
    Ring membership (0.85), chain membership (0.65), mule_score >0.5 (0.75),
    fan-out >5 (0.80)
    Input: mule_score, ring_id, chain_id, fan_out

  L6 — Velocity Burst      (weight: 0.10)
    txn_count_1h: ≥8→0.95, ≥5→0.80, ≥3→0.65; txn_count_24h ≥15→0.55

USAGE: The composite engine is called AFTER predict_single() in the PS6
score_risk tool if you want the full 6-layer breakdown. For the MVP,
predict_single() alone is sufficient.

SECTION 8: features/graph_features.py — Precomputed Graph Signals
================================================================================

FILE: features/graph_features.py
PURPOSE: Computes and caches graph-level features per account for use in
         feature engineering and model scoring.

MAIN FUNCTION: load_graph_features()
  Returns: dict {account_id: {mule_score, in_ring, passthrough_ratio,
                               is_new_account, receiver_unique_senders,
                               is_pure_receiver, is_suspected_mule}}

  This dict is the graph_features argument passed to predict_single()
  and engineer_single().

FOR PS6: In the MVP, pass graph_features=None to predict_single().
The model will still work; it just won't have the graph-derived features
(they'll all be 0). For better predictions, call load_graph_features() at
startup and pass the dict.


================================================================================
SECTION 9: HOW THE COMPONENTS CONNECT — END-TO-END FLOW
================================================================================

This is the exact data flow in FundFlow's gateway scoring pipeline.
PS6's score_risk tool replicates steps 3-7.

  1. Transaction arrives (as a dict)
  2. DB query: fetch sender's last 50 transactions from `transactions` table
     → account_history DataFrame
  3. DB query: fetch sender's KYC from `account_profiles` table
     → india_extras dict
  4. (Optional) load_graph_features() → graph_features dict
  5. Call predict_single(txn, account_history, graph_features, india_extras)
     → internally calls engineer_single() to build 48-feature vector
     → runs XGBoost model → returns fraud_probability + top_features
  6. (Optional) Call composite.py with ML score + graph signals
     → CompositeScore = 0.7×max + 0.3×weighted_avg
     → effective_score = max(ml_score, composite_score)
  7. Decision: effective_score ≥ 0.70 → BLOCKED | 0.30-0.70 → FLAGGED | else → APPROVED
  8. (FundFlow only) alert generated, stored in `alerts` table
  9. (FundFlow only) result broadcast via WebSocket to dashboard

PS6 agent replaces steps 8-9 with:
  8. Agent synthesizes transaction history + graph signals + regulation citations
  9. Agent outputs a draft STR with cited evidence

SECTION 10: config.py — ALL VARIABLES THAT MATTER FOR PS6
================================================================================

FILE: config.py (at root of the PS6 folder)

Variables PS6 uses:
  MODEL_DIR          → path to models/saved/ (where xgboost_fraud.pkl lives)
  DB_PATH            → path to fundflow.db (OVERRIDE this in PS6 via env var)
  TRANSACTION_TYPE_MAP → maps PaySim types to Indian rail names
  RISK_TIERS         → {"LOW":(0,0.3), "MEDIUM":(0.3,0.6), "HIGH":(0.6,0.8),
                         "CRITICAL":(0.8,1.0)}
  BASE_DATE          → "2026-03-01" (base date for timestamp simulation)

Variables PS6 does NOT use:
  PAYSIM_RAW_PATH, PROCESSED_DATA_PATH → for training only
  INDIAN_BRANCHES, BRANCH_CITY_MAP     → for simulation only
  CHANNELS, CHANNEL_WEIGHTS            → for simulation only
  SIMULATION_RATE, SIMULATION_FRAUD_PROBABILITY → for simulation only
  API_HOST, API_PORT, API_KEY          → FundFlow API config (PS6 has its own)

HOW TO OVERRIDE DB_PATH IN PS6 WITHOUT MODIFYING config.py:
  In your PS6 FastAPI app or run_ps6.py:
    import os
    os.environ["PS6_DB_PATH"] = "/path/to/fundflow.db"

  Then in tools.py, use:
    DB_PATH = os.environ.get("PS6_DB_PATH", "fundflow.db")

  Do NOT modify config.py's DB_PATH — it would break if config.py is imported
  from the shared location.

SECTION 12: WHAT TO DO IN PS6 ON DAY ONE
================================================================================

After copying FundFlow folder to a new PS6 directory:

  Step 1. Verify the model works:
    Run the predict_single verification command from Section 11.
    Expected: fraud_probability 0.01–0.30 for this benign transaction.

  Step 2. Point the DB at fundflow.db:
    In your .env: PS6_DB_PATH=/path/to/fundflow/fundflow.db
    Test: python inspect_db.py → should show 499,196 transactions.

  Step 3. Create agent/tools.py with the score_risk function from Section 2.

  Step 4. Create agent/tools.py get_transaction_history function:
    SQL: SELECT * FROM transactions WHERE sender_account = ? OR
         receiver_account = ? ORDER BY timestamp DESC LIMIT 50

  Step 5. Build rag/ingest.py — RAG is the only truly new infrastructure
    needed for the MVP. Everything else is wrapping existing FundFlow code.
